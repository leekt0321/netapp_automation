# Product Plan

작성일: 2026-04-16  
기준 버전 전략: `v1.0 -> v1.1 -> v1.2 -> v1.3+`

## 1. 문서 목적

이 문서는 앞으로 이 프로그램을 어떤 순서로 확장할지 정리한 실행 계획 문서다.  
현재 시스템은 이미 기본 기능이 갖춰져 있으므로, 다음 단계는 "UI를 보기 좋게 다듬어 안정적인 1.0 릴리즈를 만들고, 이후 운영 편의 기능을 단계적으로 붙여나가는 것"이다.

핵심 방향:

- `v1.0`: 지금 있는 기능을 더 보기 좋고 쓰기 편하게 정리해서 첫 안정 버전 릴리즈
- `v1.1+`: 운영자가 실제로 더 편해지는 비교/버그/문서권한/엑셀 기능 확장
- 큰 구조를 한 번에 뒤집지 않고, 현재 코드 위에서 점진적으로 기능 추가

## 2. 버전별 큰 방향

### v1.0

목표:

- 현재 기능을 유지한 채 UI 완성도 개선
- 화면 구성을 더 깔끔하게 다듬기
- 보기 좋고 사용하기 쉬운 첫 정식 릴리즈 만들기

핵심 성격:

- 기능 추가 중심이 아니라 사용성, 가독성, 시각적 정돈 중심

### v1.1

목표:

- 에러/버그를 더 체계적으로 관리할 수 있게 기능 확장
- 로그 비교 기능의 기반 도입

### v1.2

목표:

- 장비별 비교와 버그 연계를 강화
- 문서 중요도와 권한 제어를 추가

### v1.3

목표:

- summary 데이터를 외부 보고용으로 활용할 수 있게 확장
- 엑셀 다운로드 및 관리형 데이터 추출 기능 추가

## 3. v1.0 계획: UI 개선 후 첫 릴리즈

### 3.1 목표

현재 이미 구현된 기능은 다음과 같다.

- 로그인/회원가입/승인
- 사이트 관리
- 로그 업로드
- raw log 조회
- summary 조회
- manual fields 관리
- 특이사항 관리
- 수정 요청 게시판
- 버그 모음 게시판
- 회원/세션/삭제 요청 관리자 기능

`v1.0`의 목표는 이 기능들을 바꾸는 것이 아니라, 지금 상태를 더 깔끔하고 일관된 UI로 정리해서 "처음 배포해도 어색하지 않은 제품 형태"로 만드는 것이다.

### 3.2 세부 작업

#### 1. 전체 레이아웃 정리

- 대시보드, 스토리지 페이지, 버그 모음, 수정 요청 게시판, 회원 관리 화면의 간격/정렬 통일
- 카드, 패널, 테이블, 버튼 스타일 규칙 통일
- 섹션 제목과 설명 문구의 계층 구조 정리
- 모바일/좁은 화면에서 깨지는 부분 점검

#### 2. 시각적 완성도 개선

- 컬러 체계 통일
- 타이포그래피와 여백 정리
- 상태 배지, 경고, 성공, 실패 메시지 스타일 일관화
- 선택된 탭/버튼/카드가 한눈에 보이도록 강조 방식 정리

#### 3. 로그 탐색 화면 개선

- 사이트 목록, 로그 목록, 상세 영역의 정보 밀도 조정
- raw / summary 전환을 더 명확하게 표현
- summary overview와 섹션 목록의 시각적 구분 강화
- Event log 필터와 특이사항 표시를 더 읽기 쉽게 정리

#### 4. 게시판 UI 개선

- 버그 모음과 수정 요청 게시판의 검색/작성/수정 흐름 정리
- 상태값 표시 가독성 개선
- 글 목록과 상세 편집 흐름 단순화

#### 5. 관리자 화면 정리

- 회원 목록, 세션 목록, 삭제 요청 목록의 정보 구조 통일
- 승인/비활성화/삭제 요청 처리 버튼 강조 방식 정리
- 관리자 화면을 일반 운영 화면과 시각적으로 구분

### 3.3 v1.0 완료 기준

아래 기준을 만족하면 `v1.0`으로 본다.

- 현재 기능이 모두 정상 동작
- 주요 페이지가 시각적으로 통일되어 있음
- 로그 업로드부터 summary 확인까지 흐름이 자연스러움
- 게시판과 관리자 화면이 복잡하지 않게 정리됨
- "기능은 있지만 투박한 내부 툴" 느낌에서 "배포 가능한 운영 콘솔" 수준으로 올라감

## 4. 기능 로드맵

아래 항목들은 사용자께서 말씀하신 앞으로의 기능 목록을 기준으로 다시 풀어 쓴 상세 계획이다.

### 4.1 에러 모음 목록 새로 만들기

목표:

- 현재 `버그 모음`과는 별개로, 에러를 더 빠르게 모아보는 전용 목록을 만든다.

배치 방향:

- `dashboard`, `버그 모음`, `수정 요청 게시판` 흐름과 연결되는 위치에 배치
- 말씀하신 기준대로 `버그 모음`보다 위쪽에 배치

필요한 세부 기능:

- 에러 목록 전용 페이지 또는 상단 섹션 추가
- 에러 상태 분류
- 장비명, 사이트, 스토리지 기준 필터
- 최근 발생 순 정렬
- 버그 글과 연결할 수 있는 링크 구조

예상 데이터 구조:

- 별도 `error_posts` 또는 `error_records` 테이블
- 또는 초기에는 `bug_posts`를 확장하지 말고 독립 테이블로 분리하는 것이 관리상 더 명확함

화면/동작 포인트:

- "실제 장애/에러 기록"과 "개발/운영 버그 기록"을 시각적으로 구분
- 에러에서 버그로 연결, 버그에서 수정 요청으로 연결되는 흐름 고려

권장 버전:

- `v1.1`

### 4.2 같은 장비 로그 비교 후 차이점 표시

목표:

- 동일 장비의 서로 다른 시점 로그를 비교해서 무엇이 달라졌는지 보여준다.

왜 중요한가:

- 운영자는 장비 상태가 언제 어떻게 변했는지 확인하고 싶어함
- summary 값 차이를 바로 보여주면 분석 속도가 크게 올라감

필요한 세부 기능:

- 같은 장비를 식별하는 기준 정의
- 비교할 두 로그 선택 UI
- overview 주요 항목 차이 표시
- section 단위 차이 비교
- 추가/삭제/변경된 값 시각적 표시

장비 동일성 판단 후보:

- controller serial
- hostname
- model
- 수동 매핑 키

구현 방향:

- 1차는 summary key-value 기반 비교
- 2차는 section 원문 비교

표시 예시:

- ONTAP 버전 변경
- shelf 수 변경
- disk/spare 수 변경
- volume/lun 수 변경
- snapmirror 사용 여부 변경

주의점:

- 같은 장비 판별 규칙을 먼저 안정화해야 함
- 업로드 당시 추출 실패한 summary 값이 있으면 비교 품질이 떨어질 수 있음

권장 버전:

- 기초 비교 기능 `v1.1`
- 섹션별 고도화 비교 `v1.2`

### 4.3 문서 업로드 시 중요도 추가, 중요도별 삭제 권한 분리

목표:

- 업로드되는 문서/로그에 중요도를 붙이고, 중요도에 따라 삭제 권한을 다르게 적용한다.

해석 기준:

- 여기서 문서는 현재 업로드되는 로그 파일과 생성 summary를 포함하는 개념으로 보는 것이 자연스럽다.
- 필요하면 이후 일반 문서 업로드 기능으로도 확장 가능하다.

필요한 세부 기능:

- 업로드 폼에 중요도 선택 추가
- 중요도별 표시 배지 추가
- 목록 필터링
- 삭제 요청 정책 분기

예시 정책:

- 낮음: 관리자 승인 하 삭제 가능
- 보통: 관리자 승인 + 사유 필수
- 높음: 특정 관리자만 삭제 가능
- 중요: 삭제 제한 또는 2단계 확인

필요한 데이터 구조:

- `uploaded_logs`에 `importance` 컬럼 추가
- 삭제 요청 검토 시 중요도 반영
- 관리자 UI에서 중요도 표시

확장 고려:

- 중요도별 숨김/보관 정책
- 중요도별 다운로드 제한
- 중요도별 수정 가능 범위 제한

권장 버전:

- `v1.2`

### 4.4 버그 모음 페이지에서 로그를 비교해 관련 버그가 있는 장비 표시

목표:

- 버그 모음 페이지에서 단순 게시글 나열이 아니라, 특정 장비와 연관된 버그를 확인할 수 있게 한다.

핵심 개념:

- 장비 로그 비교 결과와 버그 게시글을 연결
- 특정 장비에 유사하거나 동일한 문제가 이미 기록되어 있는지 표시

필요한 세부 기능:

- 버그 글에 장비 식별자 연결
- 로그 비교 결과에서 이슈 키워드 또는 장비 정보 추출
- 버그 페이지에서 "관련 장비 있음" 표시
- 장비 상세 또는 summary에서 "연관 버그" 링크 제공

가능한 연결 방식:

- 수동 연결
- hostname / serial 기반 자동 후보 추천
- 비교 결과 기반 유사 이슈 추천

구현 순서 권장:

1. 버그 글에 장비 또는 로그 참조를 연결할 수 있게 변경
2. 특정 장비에서 관련 버그 목록 표시
3. 이후 자동 추천 또는 유사도 기반 연결 추가

권장 버전:

- 기본 연계 `v1.2`
- 자동 후보 추천 `v1.3`

### 4.5 요약개요에서 특정 정보 추출 후 엑셀 다운로드

목표:

- summary overview에 있는 핵심 항목을 모아서 엑셀로 내보낼 수 있게 한다.

사용 목적:

- 보고 자료 작성
- 장비 현황 정리
- 점검 결과 전달
- 자산/운영 정보 백업

필요한 세부 기능:

- 다운로드 대상 로그 범위 선택
- 스토리지/사이트 기준 필터
- 추출할 컬럼 선택
- 엑셀 파일 생성 및 다운로드

후보 컬럼:

- storage
- site
- filename
- hostname
- model
- serial number
- OS version
- mgmt
- shelf count
- disk/spare count
- volume count
- lun count
- manual fields 일부
- 중요도

기술 방향:

- 서버에서 xlsx 생성
- 최초 버전은 "선택된 기본 컬럼 세트 다운로드"부터 시작
- 이후 "사용자 지정 컬럼 선택"으로 확장

확장 고려:

- CSV도 같이 지원
- 특정 기간 업로드 로그 일괄 다운로드
- 비교 결과까지 별도 시트로 내보내기

권장 버전:

- 기본 다운로드 `v1.3`
- 사용자 지정 컬럼/다중 시트 `v1.3+`

## 5. 권장 구현 순서

가장 자연스러운 순서는 아래와 같다.

### 1단계: v1.0

- 기존 UI 정리
- 화면 완성도 개선
- 시각적 일관성 확보
- 첫 릴리즈

### 2단계: v1.1

- 에러 모음 목록 추가
- 같은 장비 로그 비교의 기본 기능 추가

### 3단계: v1.2

- 중요도 기반 문서/로그 관리
- 버그 모음과 장비/로그 비교 결과 연결
- 로그 비교 기능 고도화

### 4단계: v1.3

- summary overview 엑셀 다운로드
- 버그 추천/연관 기능 보강

## 6. 구현 시 공통 원칙

### 6.1 현재 구조를 최대한 활용

- 지금 시스템은 로그 업로드, summary 생성, 게시판, 관리자 기능까지 이미 동작한다.
- 따라서 새 기능은 현재 구조 위에 "연결"하는 방식이 효율적이다.

### 6.2 장비 식별 기준을 먼저 정리

로그 비교, 관련 버그 표시, 엑셀 추출 모두 장비 식별이 중요하다.

우선 후보:

- `controller_serial`
- `hostname`
- `model_name`
- 필요 시 운영자가 직접 지정하는 장비 식별 키

### 6.3 버그/에러/요청의 역할을 분리

앞으로 데이터가 늘어나면 아래 구분이 중요해진다.

- 에러 모음: 실제 운영 중 발견된 에러/장애 중심
- 버그 모음: 시스템/운영 프로세스 개선 관점의 버그 중심
- 수정 요청 게시판: 작업 요청과 진행 상태 중심

### 6.4 UI와 기능을 같이 설계

이번 로드맵의 기능들은 단순 DB 추가만으로 끝나지 않는다.

- 비교 화면
- 연관 표시
- 중요도 배지
- 다운로드 조건 선택

이런 기능은 데이터 모델과 UI를 동시에 설계해야 한다.

## 7. 최종 요약

앞으로의 방향은 명확하다.

- 먼저 현재 버전을 보기 좋게 다듬어서 `v1.0` 릴리즈
- 그 다음 운영 실무에 직접 도움이 되는 기능을 순차적으로 추가

우선순위 요약:

1. `v1.0`: UI 다듬기와 첫 안정 릴리즈
2. `v1.1`: 에러 모음 + 같은 장비 로그 비교 시작
3. `v1.2`: 중요도/권한 분기 + 버그와 장비 연결 강화
4. `v1.3`: summary 정보 엑셀 다운로드와 리포트 활용

이 문서를 기준으로 다음 구현부터는 "현재 기능 유지 + 점진적 확장" 방식으로 진행한다.
