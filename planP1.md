이 문서는 "P1. PostgreSQL 스키마 관리 정리"를 왜 지금 해야 하는지에 집중해서 정리한 문서다.  
구현 방법 자체보다, 현재 구조에서 어떤 위험이 있고 왜 P1이 다음 단계의 선행 조건이 되는지를 설명한다.

---

## 1. 결론

P1을 지금 해야 하는 가장 큰 이유는, 현재 프로젝트가 이미 PostgreSQL을 운영 DB로 사용하고 있는데도  
스키마 변경을 정식 migration 체계가 아니라 앱 startup 시점의 보정 로직에 의존하고 있기 때문이다.

즉 현재는:

- 코드가 바뀌면 DB도 같이 바뀌어야 하는데
- 그 변경 이력이 Alembic revision으로 관리되지 않고
- [app/core/lifecycle.py](/root/2026_project/app/core/lifecycle.py:37) 의 런타임 `ALTER TABLE`로 보정된다.

이 구조는 작은 프로젝트 초기에는 빠르게 움직이기 좋지만, 지금처럼

- P0 인증/인가가 이미 들어갔고
- `users`, `user_sessions`, `deletion_requests`, `uploaded_logs` 구조가 계속 커지고 있으며
- 앞으로 P2 이상에서 저장 구조를 더 손봐야 하는 단계

에서는 점점 더 위험해진다.

그래서 P1은 "깔끔함을 위한 리팩토링"이 아니라,

- 운영 DB 재현성 확보
- 배포 안정성 확보
- 이후 기능 개발 안전성 확보

를 위한 필수 작업이다.

---

## 2. 현재 구조가 왜 문제인가

### 2.1 앱이 스키마 관리까지 대신하고 있음

현재 startup에서는 다음 두 가지를 동시에 수행한다.

1. `Base.metadata.create_all(bind=engine)`
2. `ensure_schema_updates()`를 통해 누락 컬럼에 대해 직접 `ALTER TABLE` 수행

관련 코드:

- [app/core/lifecycle.py](/root/2026_project/app/core/lifecycle.py:37)

현재 여기서 보정하는 항목 예시:

- `users.role`
- `users.approved_at`
- `uploaded_logs.storage_name`
- `uploaded_logs.site_id`
- `uploaded_logs.manual_fields_json`
- `uploaded_logs.note`

이 방식의 문제는 "앱이 켜져야만 DB가 완성된다"는 점이다.

즉 지금은 배포 순서가 사실상 이렇게 된다.

1. 새 코드 배포
2. 앱 실행
3. startup에서 컬럼 확인
4. 런타임 `ALTER TABLE`
5. 그 다음부터 코드가 기대하는 스키마가 맞기 시작

이건 운영 측면에서 매우 불안정하다.

---

### 2.2 스키마 변경 이력이 남지 않는다

현재 repo에는 Alembic 골격은 있지만 실제 revision 파일이 없다.

관련 경로:

- [alembic/env.py](/root/2026_project/alembic/env.py:1)
- [alembic/README](/root/2026_project/alembic/README:1)

즉 현재는:

- 어떤 릴리즈에서 어떤 컬럼이 추가됐는지
- 운영 DB에 어떤 migration이 적용됐는지
- 특정 시점 DB 상태를 어떻게 재현해야 하는지

를 revision history로 추적할 수 없다.

문제는 이게 지금부터 더 커진다는 점이다.

P0에서 이미 추가된 개념:

- `users.role`
- `users.is_active`
- `users.approved_at`
- `user_sessions`
- `deletion_requests`

이런 변화는 더 이상 "작은 컬럼 하나 추가" 수준이 아니다.  
이미 권한 모델과 운영 흐름이 DB 구조에 직접 기대고 있다.

---

### 2.3 신규 환경 재현성이 낮다

지금 구조에서는 신규 환경에서 DB를 재현하려면 다음이 필요하다.

- 올바른 PostgreSQL 준비
- 앱 코드 버전 일치
- startup 보정 로직이 제대로 실행될 것
- 기존 컬럼 유무에 따라 동적으로 다른 `ALTER TABLE`이 돌 것

즉 DB 상태 재현이 "명시적 절차"가 아니라 "앱을 켜보면 알아서 맞춰질 것"에 가깝다.

이 방식은 아래 상황에서 특히 취약하다.

- 운영 서버를 새로 세팅할 때
- 백업 DB를 복구했을 때
- 다른 개발자가 동일 환경을 만들 때
- 릴리즈 rollback 후 다시 올릴 때

Alembic이 있어야 이게 아래처럼 바뀐다.

1. 빈 DB 준비
2. migration 적용
3. 앱 실행

이 차이는 운영성에서 매우 크다.

---

### 2.4 컬럼 추가까지만 대응하고 구조 제약은 약하다

현재 보정 로직은 대부분 "컬럼이 없으면 추가" 수준이다.

하지만 실제 운영 DB 관리에서 중요한 것은 컬럼 존재 여부만이 아니다.

예를 들면 다음이 더 중요하다.

- foreign key
- unique 제약
- index
- nullable 정책
- 기본값
- timestamp 타입 일관성
- JSON/JSONB 사용 여부

지금은 이런 스키마 품질 요소가 revision 단위로 체계적으로 관리되지 않는다.

결과적으로:

- 모델은 점점 풍부해지는데
- DB는 "일단 돌아가게" 수준으로만 확장될 위험이 있다.

이건 P2 로그/summary 저장 구조 작업으로 넘어가면 더 큰 병목이 된다.

---

## 3. 왜 하필 지금 P1인가

### 3.1 P0가 끝난 지금이 경계선이기 때문

P0 전에는 인증/권한 구조 자체가 정리되지 않았기 때문에  
DB를 먼저 다듬어도 다시 바뀔 가능성이 컸다.

하지만 지금은 P0 핵심이 정리됐다.

현재 확정된 방향:

- 서버 세션 인증
- `admin` / `user` 권한
- 관리자 승인형 회원가입
- 로그 삭제 승인 흐름
- 세션 24시간

즉 지금은 DB 차원에서 안정화해야 할 핵심 개념이 어느 정도 고정된 상태다.

그래서 지금 P1을 하면:

- 확정된 P0 구조를 migration으로 고정할 수 있고
- 이후 P2/P3/P4 작업이 이 위에서 안전하게 진행된다.

반대로 지금 P1을 미루면:

- P2에서 로그/summary 저장 구조가 더 바뀌고
- 그 변경도 또 startup 보정으로 쌓이고
- 나중에는 한번에 옮기기 더 어려워진다.

즉 지금이 "정리 가능한 마지막 좋은 시점"에 가깝다.

---

### 3.2 P2 작업이 DB 안정성을 전제로 하기 때문

P2는 로그/summary 저장 구조 재설계다.

이 단계에서 예상되는 변화:

- summary 경로 명시 저장
- 파일 저장과 DB 정합성 강화
- note/manual fields 구조 재정리
- possibly JSONB, FK, unique/index 정리

이런 작업은 migration 체계 없이 진행하면 매우 위험하다.

이유:

- 데이터 변환이 필요할 수 있고
- 기존 데이터 유지 전략이 필요하며
- 구조 변경 순서가 중요하고
- rollback 경로도 고려해야 하기 때문이다.

즉 P1은 단독 목표가 아니라, P2를 안전하게 하기 위한 기반 작업이다.

---

### 3.3 운영 릴리즈 품질을 끌어올려야 하기 때문

지금은 `0.5v`를 latest 정식 릴리즈로 확정한 상태다.  
즉 이제부터는 "실험 중인 구조"보다 "운영 가능한 구조"에 더 무게를 둬야 한다.

그런데 운영 릴리즈에서 스키마 관리가 runtime 보정에 의존하면 다음 문제가 생긴다.

- 릴리즈마다 DB 상태가 미묘하게 달라질 수 있음
- 실제 장애가 나면 원인 추적이 어려움
- 배포 문서를 쓰기 어려움
- 운영자가 앱과 DB의 책임을 분리해서 보기 어려움

정식 릴리즈 라인 이후 첫 단계가 P1이어야 하는 이유는,
"운영 가능성"을 코드 품질이 아니라 배포 재현성 관점에서 보강해야 하기 때문이다.

---

## 4. 지금 미루면 생기는 문제

### 4.1 스키마 드리프트가 계속 커진다

스키마 드리프트는 "모델이 기대하는 구조"와 "실제 DB 구조"가 점점 어긋나는 현상이다.

지금은 보정 로직이 어느 정도 따라가지만, 기능이 늘수록 이런 문제가 생긴다.

- 어떤 서버는 이미 컬럼이 있음
- 어떤 서버는 startup이 안 돌아 아직 없음
- 어떤 DB는 타입/기본값이 다름
- 어떤 컬럼은 있지만 인덱스/제약은 빠짐

이게 쌓이면 나중에 Alembic으로 전환할 때도
"기준이 되는 실제 상태"를 정의하기 어려워진다.

즉 P1은 늦출수록 쉬워지지 않고, 오히려 점점 어려워진다.

---

### 4.2 테스트와 실제 운영이 더 멀어진다

이미 테스트 체계는 약한 편이고, P3에서 복구할 계획이다.

그런데 migration 없이 스키마가 런타임에서 보정되는 구조는 테스트에도 악영향을 준다.

왜냐면 테스트는 보통:

- 고정된 스키마
- 고정된 초기 데이터
- 재현 가능한 DB 상태

를 전제로 해야 하는데, 지금은 startup 동작에 따라 DB 상태가 달라질 수 있기 때문이다.

즉 P1이 끝나야 P3 테스트 복구도 더 의미 있게 할 수 있다.

---

### 4.3 운영 장애 시 복구 절차가 모호해진다

운영 중 DB 문제가 생겼을 때 중요한 질문은 보통 이렇다.

- 어떤 migration까지 적용됐는가
- 어떤 릴리즈가 어떤 스키마를 기대하는가
- 롤백하려면 어떤 상태로 돌아가야 하는가

지금은 이 질문에 revision 번호로 답할 수 없다.

그 결과:

- 복구 절차가 사람 기억에 의존하게 되고
- 운영 리스크가 커진다.

P1은 결국 "운영 문서화 가능성"을 높이는 작업이기도 하다.

---

## 5. P1이 해결하려는 핵심 문제

P1은 단순히 Alembic을 쓰자는 얘기가 아니다.  
실제로는 아래 문제를 해결하려는 단계다.

1. 앱 startup이 DB 구조를 몰래 바꾸는 문제
2. 스키마 변경 이력이 없는 문제
3. 신규 환경/복구 환경 재현성이 낮은 문제
4. 컬럼 중심 보정만 있고 제약/index 관리가 약한 문제
5. 다음 단계 저장 구조 개편을 안전하게 못 하는 문제

즉 P1의 본질은:

- schema ownership을 startup 코드에서 migration 체계로 옮기는 것

이다.

---

## 6. P1이 끝나면 얻는 것

P1이 끝나면 기대할 수 있는 효과는 명확하다.

### 운영 측면

- 신규 서버 세팅 절차가 단순해짐
- 복구 절차가 명확해짐
- 릴리즈별 DB 요구사항을 추적 가능

### 개발 측면

- 모델 변경 시 revision 단위로 관리 가능
- 구조 변경 작업이 더 예측 가능해짐
- 이후 기능 개발이 "DB가 알아서 맞춰지겠지"에 기대지 않게 됨

### 테스트 측면

- 테스트 DB를 더 일관되게 만들 수 있음
- fixture와 스키마 상태를 명시적으로 맞출 수 있음

### 다음 단계 연계

- P2 저장 구조 개편 안전성 상승
- P3 테스트 복구 기반 확보
- P6 운영성 강화 작업과 자연스럽게 연결

---

## 7. P1은 무엇을 하지 않는가

P1의 범위를 분명히 해야 한다.

P1은 아래를 주로 다룬다.

- Alembic 실사용 시작
- startup `ALTER TABLE` 제거
- 테이블/컬럼/제약/index를 migration으로 관리

반면 아래는 P1의 핵심 범위가 아니다.

- 로그 summary 비즈니스 로직 재설계
- 프론트 대규모 구조 분리
- 성능 튜닝 전반
- 게시판 UX 개선

이 구분이 필요한 이유는, P1이 너무 커지면 migration 정착이라는 가장 중요한 목표를 놓치기 쉽기 때문이다.

## 8. 현재 코드 기준 P1 필요성을 가장 잘 보여주는 파일

### [app/core/lifecycle.py](/root/2026_project/app/core/lifecycle.py:1)

이 파일이 P1의 가장 직접적인 근거다.

이유:

- startup에서 schema 보정 수행
- runtime `ALTER TABLE` 직접 실행
- 앱 실행과 DB 구조 변경이 결합되어 있음

즉 P1이 끝나면 이 파일의 schema 보정 책임은 크게 줄어들어야 한다.

### [app/models.py](/root/2026_project/app/models.py:1)

이 파일은 현재 앱이 기대하는 데이터 구조를 표현한다.  
하지만 이 기대가 migration 이력과 아직 1:1로 연결되어 있지 않다.

P1은 모델 선언과 실제 DB 상태를 revision 체계로 연결하는 작업이다.

### [plan.md](/root/2026_project/plan.md:539)

이미 전체 계획에서도 P1은
"PostgreSQL 스키마 관리 정리"로 별도 우선순위를 차지하고 있다.

이건 단지 tidy-up이 아니라 다음 단계 전체의 기반이라는 뜻이다.

## 9. 최종 판단

P1을 지금 해야 하는 이유는 명확하다.

- P0가 끝나면서 핵심 권한 구조가 고정됐다.
- 하지만 DB는 아직 runtime 보정에 의존한다.
- 이 상태로 P2 이후를 진행하면 스키마 드리프트와 운영 리스크가 더 커진다.
- Alembic 전환은 늦출수록 쉬워지지 않는다.

따라서 P1은 "하면 좋은 작업"이 아니라,

- 현재 latest 릴리즈 이후
- 다음 버전 개발을 안전하게 시작하기 위한
- 첫 번째 정리 작업

으로 봐야 한다.

한 줄로 요약하면:

P1은 지금 프로젝트에서 "DB를 코드에 맞춰 자동으로 얼기설기 보정하는 단계"를 끝내고,  
"스키마를 명시적으로 버전 관리하는 단계"로 넘어가기 위해 반드시 필요한 전환 작업이다.
