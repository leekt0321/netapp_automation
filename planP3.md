이 문서는 "P3. 테스트 체계 복구"를 왜 지금 해야 하는지에 집중해서 정리한 문서다.  
핵심은 테스트를 단순 보조 수단으로 추가하는 것이 아니라, 지금까지 정리한 인증/권한, migration, 저장 구조 변경이 이후 단계에서 다시 깨지지 않도록 회귀 방어막을 세우는 데 있다.

---

## 1. 결론

P3를 지금 해야 하는 이유는, 현재 프로젝트가 이미

- 인증/인가 흐름
- PostgreSQL/Alembic 스키마 관리
- 로그/summary 저장 구조

까지 손을 댄 상태인데, 이 변경들을 자동으로 검증해 줄 안전망이 매우 약하기 때문이다.

지금 시스템은 기능 수가 적지 않다.

- 회원가입/로그인/권한
- 사이트 관리
- 로그 업로드/목록/원본/summary
- 수동 입력 필드/특이사항
- 삭제 요청 승인
- 게시판/버그 게시판

하지만 이 기능들에 대해 "지금은 돌아간다"와 "앞으로도 안 깨진다"는 전혀 다른 문제다.

현재 단계에서는 사람이 직접 눌러서 확인하는 수동 검증에 많이 기대고 있고,  
이 방식은 기능 수가 늘어날수록 빠르게 한계가 온다.

그래서 P3의 목적은 다음 한 줄로 요약할 수 있다.

- 지금까지 만든 중요한 동작을 자동 검증 가능한 상태로 바꿔, 이후 작업 속도와 운영 신뢰성을 동시에 확보하는 것

즉 P3는 품질 향상 옵션이 아니라,

- P4 프론트 구조 분리
- P5 UX 개선
- P6 운영성 강화

를 안전하게 진행하기 위한 기반 작업이다.

## 2. 현재 테스트 상태가 왜 문제인가

### 2.1 기존 테스트가 현행 코드와 어긋나 있다

[research.md](/root/2026_project/research.md:592) 에서도 정리되어 있듯이, 현재 테스트 파일은 현행 코드와 맞지 않는 흔적이 많다.

대표적으로:

- 과거 SQLite 기준 흔적이 남아 있음
- 현재 API/서비스 구조와 불일치 가능성이 큼
- 현재 저장 구조 및 인증 구조를 충분히 반영하지 못함

즉 "테스트가 있다"는 사실보다 중요한 것은
"그 테스트를 믿을 수 있느냐"인데, 지금은 그 신뢰도가 낮다.

이 상태에서는 테스트가 회귀 방어막이 아니라,
오히려 오래된 가정의 집합이 될 수 있다.

---

### 2.2 지금 중요한 변경들이 테스트 없이 누적됐다

P0, P1, P2에서 이미 상당히 중요한 변경이 들어갔다.

- P0
  - 서버 세션 기반 인증
  - 관리자 승인형 회원가입
  - 로그 삭제 승인 흐름
  - 게시판 권한 정책
- P1
  - Alembic revision 관리 시작
  - startup 런타임 스키마 보정 제거
  - migration 선적용 전제 도입
- P2
  - `uploaded_logs.summary_path` 추가
  - 업로드 중간 실패 시 파일 cleanup 보강
  - summary 조회/삭제를 DB 경로 기준으로 변경

이 세 단계는 전부 "깨지면 바로 사용자 경험이나 운영 데이터에 영향이 가는 변경"이다.

하지만 지금은 이 동작을 항상 자동 검증해주는 테스트 세트가 없다.

즉 현재 문제는 단순히 "테스트가 적다"가 아니라,

- 중요한 구조 변경은 이미 많고
- 회귀 검증은 약한 상태

라는 점이다.

---

### 2.3 수동 확인만으로는 이제 커버가 어렵다

지금까지는 주로 아래 방식으로 확인해왔다.

- 브라우저에서 실제 클릭 확인
- `TestClient`로 일부 흐름 점검
- `py_compile` 확인
- `alembic check` 확인

이 방식은 초기에는 충분할 수 있다.

하지만 지금 프로젝트는 이미 단일 기능 앱이 아니라
여러 흐름이 연결된 운영 도구다.

예를 들면:

- 회원가입 후 승인 대기 -> 관리자 승인 -> 로그인 가능
- 로그 업로드 -> summary 생성 -> raw/summary 조회
- 삭제 요청 생성 -> 관리자 검토 -> 실제 삭제
- 일반 사용자/관리자별 UI/권한 차이

이런 흐름은 하나씩 수동으로 확인하면 시간이 오래 걸리고,
작은 수정이 여러 기능에 미치는 영향도 놓치기 쉽다.

즉 이제는 "동작 확인"을 사람 기억에 의존하면 안 되는 단계다.

## 3. 왜 지금 P3인가

### 3.1 P1, P2가 끝난 지금이 테스트 기준선을 세우기 가장 좋다

테스트는 너무 이르게 시작해도 유지 비용이 크고,
너무 늦게 시작해도 이미 불안정한 구조를 그대로 따라가게 된다.

지금은 그 중간에서 가장 좋은 시점에 가깝다.

현재 상태:

- P0로 권한 경계가 어느 정도 정리됨
- P1로 DB 스키마 관리 방식이 정리됨
- P2로 로그/summary 저장 기준이 더 명확해짐

즉 이제야 테스트가 기대값을 안정적으로 정의할 수 있는 상태가 됐다.

예를 들면 지금은 아래를 명확히 테스트할 수 있다.

- `alembic upgrade head` 이후 앱 startup 가능 여부
- 승인 전/후 로그인 정책
- 로그 업로드 시 raw/summary/DB 상태
- summary 조회가 DB 경로 기준으로 동작하는지
- 삭제 후 raw/summary/DB가 일관되게 정리되는지

P1/P2 이전에는 이런 기대값 자체가 흔들릴 수 있었지만,
지금은 기준선을 세울 수 있다.

---

### 3.2 P4, P5로 가기 전에 회귀 방어막이 필요하다

다음 단계 후보를 보면:

- P4: 프론트 구조 분리
- P5: UX 개선

둘 다 "코드를 많이 건드리지만, 사용자 입장에선 기존 기능도 그대로 살아 있어야 하는" 작업이다.

테스트가 없으면 이런 작업은 매번 불안해진다.

예를 들면 프론트 분리를 하다가도:

- 로그인 복원 흐름
- 권한별 노출 차이
- 삭제 요청 버튼 표시 조건
- summary 응답 형태

같은 게 조용히 깨질 수 있다.

즉 P3는 다음 기능 작업의 속도를 늦추는 단계가 아니라,
오히려 이후 변경 속도를 올리는 단계다.

---

### 3.3 운영 문제를 "재현 가능"하게 만들어 준다

현재 운영 리스크의 큰 문제는 문제가 생겼을 때
"다시 만들어서 확인하기 어렵다"는 점이다.

예를 들면:

- 특정 로그에서 parser가 깨지는지
- 어떤 업로드 실패에서 파일 찌꺼기가 남는지
- 승인 대기 계정 정책이 정확히 맞는지

를 사람이 매번 수동으로 재현하는 건 비용이 크다.

테스트가 있으면 이런 문제를

- fixture
- 샘플 로그
- 분리된 test DB

를 이용해 반복 재현 가능하게 만들 수 있다.

이건 단순한 품질 문제가 아니라,
운영 이슈 대응 속도와 직접 연결된다.

---

## 4. 지금 미루면 생기는 문제

### 4.1 이후 변경이 전부 더 불안해진다

테스트 없이 P4, P5, P6를 계속 진행하면
이후 변경은 전부 아래 리스크를 안고 가게 된다.

- 수정 후 무엇이 깨졌는지 늦게 발견
- 릴리즈 직전에 수동 점검 범위가 계속 증가
- 고친 버그가 다른 곳을 다시 깨뜨림
- "건드리기 무서운 코드"가 늘어남

즉 테스트 부재는 시간이 갈수록 개발 속도를 떨어뜨린다.

---

### 4.2 parser와 저장 구조 변경의 위험이 계속 커진다

이 프로젝트에서 특히 테스트가 중요한 영역은 두 군데다.

1. parser
- 입력 포맷 변화에 민감함
- 장비별 로그 변형 가능성이 큼
- summary 결과가 화면과 운영 판단에 직접 연결됨

2. 로그 저장 구조
- 파일과 DB가 함께 움직임
- 업로드/삭제 실패 시 정합성 이슈가 생길 수 있음

이 두 영역은 눈으로 대충 확인해서 넘기기 어렵다.

즉 테스트가 늦어질수록:

- parser는 더 깨지기 쉬워지고
- 저장 구조는 더 검증하기 어려워진다.

---

### 4.3 문서와 실제 동작이 다시 벌어질 수 있다

지금까지 `research.md`, `research2.md`, `plan.md`, `planP1.md`, `planP2.md`까지 문서가 꽤 잘 쌓였다.

하지만 테스트가 없으면
"문서상 기대하는 구조"와 "실제 코드 동작"이 다시 벌어질 가능성이 있다.

특히 지금처럼 내부 운영 도구는,
문서보다 실제 동작이 더 빨리 변한다.

그래서 P3는 단순히 코드 검증이 아니라,

- 문서화된 정책
- 실제 구현
- 실제 동작 결과

를 더 가깝게 붙이는 역할도 한다.

---

## 5. P3가 해결하려는 핵심 문제

P3가 해결하려는 핵심은 아래 다섯 가지다.

1. 오래된 테스트와 현재 코드 사이의 불일치
2. 인증/권한 핵심 흐름에 대한 자동 검증 부재
3. 로그 업로드/summary/삭제 흐름의 회귀 검증 부재
4. parser 결과에 대한 안정적인 fixture 기반 검증 부재
5. 릴리즈 전 품질 확인이 사람 손에 과도하게 의존하는 문제

즉 P3의 본질은:

- "중요한 기능을 테스트 가능한 계약으로 다시 정의하는 것"

이다.

## 6. P3가 끝나면 얻는 것

### 운영 측면

- 릴리즈 전에 최소 회귀 세트를 자동으로 돌릴 수 있음
- 인증/로그/summary 핵심 장애를 더 빨리 발견 가능
- 운영 이슈 재현과 재검증이 쉬워짐

### 개발 측면

- 리팩토링에 대한 심리적 비용 감소
- 변경 후 영향 범위 파악이 빨라짐
- parser, auth, log_service 같은 핵심 모듈을 더 자신 있게 수정 가능

### 품질 측면

- "우연히 된다"가 아니라 "기대한 대로 동작한다"를 반복 확인 가능
- 버그 수정 후 재발 방지 가능
- pre-release와 final release 간 품질 차이를 더 명확히 만들 수 있음

### 다음 단계 연계

- P4 프론트 구조 분리의 안전성 증가
- P5 UX 개선 속도 증가
- P6 운영 점검 자동화와 연결 쉬움

---

## 7. P3는 무엇을 하지 않는가

P3 범위를 분명히 할 필요가 있다.

P3는 아래를 주로 다룬다.

- 깨진 테스트 폐기 또는 재작성
- parser 단위 테스트
- auth/service/API 통합 테스트
- PostgreSQL 기준 test fixture 정리
- 최소 릴리즈 회귀 세트 정의

반면 아래는 P3의 핵심 목표가 아니다.

- 프론트 E2E 테스트 전면 도입
- 성능 테스트 체계 구축
- 부하 테스트
- CI/CD 파이프라인 전면 정비
- 모니터링 시스템 도입

즉 P3는 "모든 테스트를 한 번에 다 만드는 단계"가 아니라,
지금 가장 중요한 기능을 자동 검증 가능한 수준까지 끌어올리는 단계다.

## 8. 현재 코드 기준 P3 필요성을 가장 잘 보여주는 파일

### [tests/conftest.py](/root/2026_project/tests/conftest.py:1)

이 파일은 P3 필요성을 가장 직접적으로 보여준다.

이유:

- 테스트 앱 구동 기준
- 테스트용 DB/업로드 디렉토리 분리
- 역할별 테스트 파일의 공통 기반

이 세 가지를 한 군데에서 정리하고 있기 때문이다.

즉 P3는 "테스트가 없는 상태에서 새로 만드는 것"만이 아니라,
"분산된 테스트 전제를 현재 구조에 맞게 다시 세우는 것"도 포함한다.

### [app/services/log_service.py](/root/2026_project/app/services/log_service.py:1)

P2로 이 파일의 저장 구조는 더 좋아졌지만,
그만큼 이제는 이 기대값을 테스트로 고정해야 한다.

특히 중요한 포인트:

- summary 파일 생성
- `summary_path` 저장
- 실패 시 cleanup
- 삭제 시 raw/summary 정리
- summary 조회

이런 흐름은 테스트 없이는 다시 쉽게 흔들릴 수 있다.

### [app/services/auth_service.py](/root/2026_project/app/services/auth_service.py:1)

현재 인증 정책은 예전보다 훨씬 복잡해졌다.

- 승인 대기 계정
- 활성/비활성
- 세션 24시간
- 권한별 응답 차이

즉 auth는 지금부터는 반드시 자동 검증이 필요한 영역이다.

### [app/parser_netapp.py](/root/2026_project/app/parser_netapp.py:1)

parser는 이 프로젝트에서 가장 입력 변화에 민감한 영역 중 하나다.

장비 로그 포맷이 조금만 바뀌어도 summary 결과가 달라질 수 있기 때문에,
fixture 기반 단위 테스트가 없으면 앞으로 안정성을 유지하기 어렵다.

## 9. 최종 판단

P3를 지금 해야 하는 이유는 명확하다.

- P0로 접근 권한을 정리했고
- P1로 스키마 관리 체계를 세웠고
- P2로 저장 구조의 핵심 위험을 낮췄다.

이제 남은 핵심 과제는
"이 상태를 계속 안전하게 유지할 수 있느냐"다.

테스트가 없으면:

- 지금까지 정리한 구조를 자신 있게 건드릴 수 없고
- 다음 단계 작업 속도가 느려지고
- 릴리즈 품질도 계속 수동 점검에 의존하게 된다.

따라서 P3는 단순한 품질 개선이 아니라,

- 지금까지 만든 기반을 실제 개발 생산성으로 바꾸는 단계

라고 보는 것이 맞다.

한 줄로 요약하면:

P3는 지금 프로젝트에서 "중요한 기능이 우연히 돌아가는 상태"를 끝내고,  
"중요한 기능이 반복해서 검증되는 상태"로 넘어가기 위해 반드시 필요한 단계다.

===============

## 진행 후 해결된 부분

P3 1차 작업을 통해 아래 항목이 실제 코드 수준에서 정리되었다.

### 1. 오래된 테스트를 현재 구조 기준으로 재구성함

기존 단일 테스트 파일은
현재 앱 구조와 맞지 않는 가정이 많았고, 단일 파일에 여러 책임이 섞여 있었다.

지금은 이를 정리해서 아래처럼 역할별 테스트로 분리했다.

- [tests/conftest.py](/root/2026_project/tests/conftest.py:1)
- [tests/test_auth.py](/root/2026_project/tests/test_auth.py:1)
- [tests/test_sites.py](/root/2026_project/tests/test_sites.py:1)
- [tests/test_logs.py](/root/2026_project/tests/test_logs.py:1)
- [tests/test_parser_netapp.py](/root/2026_project/tests/test_parser_netapp.py:1)

즉 "예전 단일 테스트 파일"에서
"현재 기능 책임에 맞는 테스트 묶음"으로 재편되었다.

### 2. 테스트 앱 실행 기준을 분리된 환경으로 고정함

[tests/conftest.py](/root/2026_project/tests/conftest.py:1) 에서

- 테스트용 `DATABASE_URL`
- 테스트용 `UPLOAD_DIR`
- 테스트용 관리자 계정

을 별도로 주입하도록 만들었다.

또한:

- `TEST_DATABASE_URL` 환경변수가 있으면 전용 테스트 DB 사용
- 없으면 임시 SQLite DB 사용

으로 구성해서 운영 DB를 건드리지 않는 기본 실행 경로를 만들었다.

즉 이제 테스트는 운영 상태에 덜 의존하고 반복 실행 가능한 형태가 되었다.

### 3. 인증/승인 핵심 흐름에 대한 회귀 테스트가 생김

[tests/test_auth.py](/root/2026_project/tests/test_auth.py:1) 에서 아래 동작을 자동 검증한다.

- 회원가입 후 승인 대기 상태 생성
- 승인 전 로그인 차단
- 관리자 승인 후 로그인 가능
- 비밀번호 변경
- 비밀번호 변경 후 이전 비밀번호 로그인 실패

즉 P0에서 만든 핵심 인증 정책이
"수동 확인"이 아니라 "자동 검증" 대상이 되었다.

### 4. 사이트 권한 정책을 자동 검증하게 됨

[tests/test_sites.py](/root/2026_project/tests/test_sites.py:1) 에서 아래를 검증한다.

- 관리자는 사이트 생성 가능
- 일반 사용자는 사이트 목록 조회 가능
- 일반 사용자는 사이트 생성 불가

즉 스토리지 사이트 관련 권한 정책도 회귀 보호 범위 안으로 들어왔다.

### 5. 게시판 공용 수정/삭제 정책도 자동 검증하게 됨

[tests/test_boards.py](/root/2026_project/tests/test_boards.py:1) 에서 아래를 검증한다.

- 일반 사용자 로그인
- 수정 요청 게시판 생성/수정/조회/삭제
- 버그 게시판 생성/수정/조회/삭제
- 작성자 표시가 현재 사용자 기준으로 들어가는지

즉 P0에서 확정한 "일반 사용자 공용 게시판" 정책도 회귀 보호 범위에 들어왔다.

### 6. 로그 업로드/summary/삭제 승인 흐름을 한 번에 검증하게 됨

[tests/test_logs.py](/root/2026_project/tests/test_logs.py:1) 에서 아래 흐름을 연결해서 검증한다.

- 사이트 생성
- 일반 사용자 승인
- 로그 업로드
- raw/summary 파일 생성 확인
- 로그 목록/summary 응답 확인
- 삭제 요청 생성
- 관리자 승인
- raw/summary 파일 정리 확인

즉 P2에서 정리한 저장 구조와 P0의 삭제 승인 정책이
실제 API 흐름 기준으로 검증 가능해졌다.

### 7. 관리자 운영 기능도 자동 검증하게 됨

[tests/test_admin.py](/root/2026_project/tests/test_admin.py:1) 에서 아래를 검증한다.

- 관리자 세션 목록 조회
- 일반 사용자 로그인 세션이 목록에 잡히는지
- 사용자 비활성화 시 해당 사용자의 세션이 무효화되는지
- 로그 삭제 요청 reject 시 파일이 유지되는지

즉 관리자 운영 기능도 단순 수동 확인 범위를 넘어 자동 검증 대상이 되었다.

### 8. 예외/실패 경로 테스트가 추가됨

[tests/test_logs.py](/root/2026_project/tests/test_logs.py:1) 에서 아래 실패 시나리오도 검증한다.

- 잘못된 스토리지/사이트 조합 업로드 차단
- summary 파일 유실 시 404 응답
- parser 실패 시 업로드 파일 cleanup

즉 이번 P3로 "성공 경로만 확인하는 테스트"에서
"실패 시 기대 동작도 검증하는 테스트"로 한 단계 더 나아갔다.

### 9. parser 핵심 결과에 대한 단위 테스트가 추가됨

[tests/test_parser_netapp.py](/root/2026_project/tests/test_parser_netapp.py:1) 에서

- 단일 클러스터 샘플
- 12노드 샘플

에 대해 hostname, model, serial 같은 핵심 추출 결과를 검증한다.

즉 parser는 이제 적어도 핵심 샘플 기준으로
출력이 갑자기 바뀌는 회귀를 잡을 수 있게 됐다.

### 10. 현재 1차 회귀 세트는 실제로 통과함

검증 결과:

- `.venv` 기준 `pytest` 실행 성공
- 총 `12 passed`

즉 P3 1차 작업은 단순 scaffold 수준이 아니라,
실제로 돌아가는 최소 회귀 세트를 만든 상태라고 볼 수 있다.

### 11. 현재 판단

현재 기준으로 보면 P3의 핵심 목표는 마감 가능한 수준까지 도달했다.

이미 확보된 것:

- 테스트 실행 기반
- 역할별 테스트 파일 분리
- auth/site/log/board/admin/parser 핵심 흐름 검증
- 성공/실패 경로를 함께 포함한 최소 회귀 세트
- 실제 실행 결과 `12 passed`

즉 이번 P3는 "핵심 흐름 기준의 최소 회귀 세트 복구"를 넘어서,
"주요 성공/실패 경로까지 포함한 1차 회귀 체계 정리"까지 완료된 상태로 보는 것이 정확하다.

### 12. 추후 보강 가능 항목

아래는 P3의 미완료라기보다, 다음 단계에서 계속 보강할 수 있는 확장 항목이다.

- PostgreSQL 전용 test DB를 기본 전략으로 더 강하게 고정
- 더 세밀한 parser 변형 케이스 fixture 추가
- 게시판/관리자 기능의 edge case 테스트 확대
- CI 자동 실행 연결

여기서 CI는 현재 필수 범위가 아니라, 추후 고려 사항으로만 남겨두는 것이 맞다.
