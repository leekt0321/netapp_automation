이 문서는 "P6. 운영성/관측성 강화"를 왜 지금 해야 하는지에 집중해서 정리한 문서다.  
핵심은 새 기능을 더 넣는 것이 아니라, 장애를 더 빨리 발견하고, 데이터 이상을 더 쉽게 추적하고, 운영 작업을 더 덜 불안하게 만드는 데 있다.

---

## 1. 결론

P6를 지금 해야 하는 이유는, 현재 프로젝트가

- 기능적으로는 꽤 성숙했고
- 기반 구조도 많이 정리됐지만
- 실제 운영 중 무슨 일이 일어나는지 파악하는 수단은 아직 약하기 때문이다.

지금은 이미 다음이 가능하다.

- 인증/권한 관리
- migration 기반 스키마 관리
- 로그/summary 저장 정합성 개선
- 테스트 기반 회귀 검증
- 프론트 구조 분리

즉 이제부터 남는 큰 문제는
"장애나 이상 상태가 생겼을 때 얼마나 빨리 알 수 있고, 얼마나 빨리 원인을 좁힐 수 있느냐"다.

그래서 P6의 목적은 다음 한 줄로 요약할 수 있다.

- 시스템이 잘못될 때 더 빨리 보이고, 더 빨리 확인하고, 더 빨리 복구할 수 있게 만드는 것

즉 P6는 기능 개발 단계가 아니라

- 관측성
- 점검 가능성
- 복구 준비도

를 올리는 운영 단계다.

---

## 2. 현재 운영성이 왜 부족한가

### 2.1 장애를 구조적으로 추적하기 어렵다

지금도 로그 업로드 실패, summary 누락, 파싱 실패, 삭제 실패 같은 이슈는 발생할 수 있다.

하지만 현재 구조에서는:

- 어디서 실패했는지
- 어떤 파일/로그가 문제인지
- DB와 파일 상태가 어떻게 어긋났는지

를 한눈에 보기 어렵다.

즉 지금은 "문제가 생기면 직접 들어가서 찾아보는 운영"에 가깝고,
"문제가 생기면 바로 보이는 운영"은 아니다.

---

### 2.2 health는 있어도 운영 판단 정보는 부족할 수 있다

[research2.md](/root/2026_project/research2.md:76) 에서도 `/health`가 언급되지만,
실제 운영에서 필요한 건 단순 생존 확인을 넘어선 정보다.

예를 들면:

- DB 연결 가능한지
- 업로드 디렉토리에 접근 가능한지
- migration 상태가 맞는지
- orphan 파일/레코드 가능성이 있는지

즉 health도 더 운영 친화적인 형태로 확장될 여지가 있다.

---

### 2.3 파일-DB 정합성은 좋아졌지만, 점검 도구는 아직 없다

P2에서 정합성은 많이 좋아졌다.

- `summary_path` 저장
- 업로드/삭제 cleanup 보강
- DB 기준 처리 강화

하지만 운영자 관점에서는 여전히 필요하다.

- 지금 DB에는 있는데 파일이 없는 로그가 있는지
- 파일은 남았는데 DB에는 없는 orphan이 있는지
- summary가 비어 있는 레코드가 있는지

즉 정합성 보강과 정합성 점검은 다른 문제다.

P2가 저장 구조를 개선했다면,
P6는 그 구조를 운영 중 점검하는 도구를 만드는 단계다.

---

## 3. 왜 지금 P6인가

### 3.1 기반 정리가 끝나야 운영 도구도 의미가 있다

P6는 앞 단계가 정리된 뒤에야 가치가 커진다.

왜냐면:

- P1 이전에는 스키마 기준이 흔들렸고
- P2 이전에는 저장 구조 기준이 흔들렸고
- P3 이전에는 회귀 기준이 약했고
- P4 이전에는 프론트 구조도 비대했다

이 상태에서 운영 도구를 먼저 만들면
"불안정한 구조를 관찰하는 도구"만 만들게 될 위험이 있었다.

지금은 다르다.

- 스키마 기준 있음
- 저장 기준 있음
- 테스트 기준 있음

즉 이제 운영성 강화가 실제로 의미를 가지는 시점이다.

---

### 3.2 시스템이 커질수록 운영성 부족이 더 아프다

운영성 문제는 기능이 적을 때는 잘 드러나지 않지만,
시스템이 커질수록 갑자기 아파진다.

예를 들면:

- 업로드 로그 수 증가
- 사이트 수 증가
- 사용자 수 증가
- 게시판 글 수 증가

이 상황에서 관측성과 점검 도구가 없으면,
장애 하나를 확인하는 데 드는 시간이 급격히 늘어난다.

즉 P6는 "나중에 필요할 것"이 아니라,
시스템이 더 커지기 전에 미리 준비하는 단계에 가깝다.

---

## 4. 지금 미루면 생기는 문제

### 4.1 문제가 생겨도 원인을 찾는 시간이 길어진다

운영에서 가장 비싼 건 장애 그 자체보다,
"왜 그랬는지 모르는 시간"인 경우가 많다.

관측성이 부족하면:

- 재현까지 오래 걸리고
- 데이터 확인이 분산되고
- 대응이 사람 기억과 수동 점검에 의존하게 된다

즉 미루면 장애 대응 비용이 커진다.

---

### 4.2 정합성 문제를 늦게 발견하게 된다

파일/DB 정합성 문제는 초기에 바로 잡으면 작지만,
늦게 발견하면 누적된 데이터 때문에 커진다.

예:

- orphan 파일 다수 누적
- summary 없는 로그 누적
- 삭제 후 잔여 파일 누적

즉 점검 도구가 없으면 "조용한 이상 상태"를 늦게 보게 된다.

---

## 5. P6가 해결하려는 핵심 문제

P6가 해결하려는 핵심은 아래 다섯 가지다.

1. 장애/실패 상황의 낮은 가시성
2. health 정보의 운영 활용도 부족
3. 파일-DB 정합성 점검 도구 부재
4. orphan 상태 탐지/복구 준비 부족
5. 운영자가 상태를 판단할 기준의 부족

즉 P6의 본질은:

- "시스템을 고치는 능력보다, 시스템 상태를 빨리 파악하는 능력을 높이는 것"

이다.

---

## 6. P6가 끝나면 얻는 것

### 운영 측면

- 장애 탐지 속도 향상
- 로그/파일/DB 이상 상태 파악 쉬움
- 정합성 점검 작업 반복 가능

### 개발 측면

- 문제 재현과 원인 축소가 빨라짐
- 운영 이슈를 코드 수정으로 연결하기 쉬워짐

### 조직 측면

- 특정 사람 기억에 덜 의존하게 됨
- 운영 인수인계가 쉬워짐
- 릴리즈 후 점검 기준이 더 명확해짐

---

## 7. P6는 무엇을 하지 않는가

P6 범위를 분명히 할 필요가 있다.

P6는 아래를 주로 다룬다.

- 구조화 로그/실패 로그 보강
- health check 확장
- 파일/DB 정합성 점검 스크립트
- 운영 문서 보강

반면 아래는 P6의 핵심 목표가 아니다.

- 대규모 모니터링 플랫폼 도입
- 분산 추적 시스템 도입
- 성능 튜닝 자체
- 새로운 비즈니스 기능 추가

즉 P6는 "과한 운영 플랫폼 구축"이 아니라,
"현재 시스템에 맞는 실용 운영성 확보" 단계다.

## 진행 후 해결된 부분

2026-04-16 기준 P6 1차 구현으로, 이전까지 흩어져 있던 운영 확인 지점을 실제로 사용할 수 있는 형태로 정리했다.

### 1. `/health`가 단순 생존 확인에서 운영 상태 요약으로 바뀌었다

이전 `/health`는 사실상

- 앱 응답 여부
- DB `SELECT 1`

정도만 확인하는 매우 얇은 엔드포인트였다.

지금은 다음을 함께 본다.

- DB 연결 상태
- 업로드 디렉토리 존재 여부
- 업로드 디렉토리 writable 여부
- Alembic revision 조회 가능 여부
- 필수 테이블 누락 여부
- 전체 상태 `ok/degraded`

즉 운영자가 가장 먼저 보는 확인 지점이
"살아 있나?"에서
"어느 층이 문제인가?"까지 좁힐 수 있는 형태가 됐다.

### 2. 운영자가 확인할 수 있는 정합성 점검 보고서가 생겼다

이번에 파일/DB 정합성을 직접 점검하는 로직을 추가했다.

확인 가능한 항목:

- DB에는 있는데 원본 파일이 없는 로그
- DB에는 있는데 summary 파일이 없는 로그
- upload 디렉토리에 남아 있지만 DB에는 없는 orphan raw 파일
- upload 디렉토리에 남아 있지만 DB에는 없는 orphan summary 파일
- DB가 가리키는 경로가 upload 디렉토리 밖으로 벗어난 로그

즉 P2가 "정합성이 덜 깨지게" 만들었다면,
P6는 "정합성이 깨졌는지 운영자가 직접 확인하게" 만들었다.

### 3. 관리자 확인 지점이 하나 더 생겼다

이전에는 정합성 확인을 하려면

- 직접 DB를 보고
- upload 디렉토리를 보고
- 파일 유무를 손으로 맞춰봐야 했다.

지금은 관리자 권한으로

- `/admin/operations/integrity`

를 호출하면 현재 상태를 바로 확인할 수 있다.

즉 운영 중 이상이 생겼을 때
"어디서부터 볼까?"가 아니라
"이 엔드포인트부터 보면 된다"는 기준점이 생겼다.

### 4. CLI 점검 스크립트가 생겼다

웹이나 브라우저를 거치지 않고도 운영 점검을 할 수 있게

- `scripts/check_integrity.py`

를 추가했다.

이 스크립트는

- 기본 텍스트 출력
- `--json`
- `--fail-on-issues`

를 지원해서, 수동 점검과 자동 점검 둘 다에 쓸 수 있다.

즉 운영자 입장에서
"웹으로 확인"과
"터미널에서 바로 확인" 두 경로가 생긴 셈이다.

### 5. 업로드/요약 조회/삭제 실패를 로그로 더 빨리 좁힐 수 있게 됐다

로그 서비스에 운영 로그를 보강해서 다음 상황이 남는다.

- 로그 업로드 성공
- 로그 업로드 중 HTTP/DB/예외 실패
- 원본 로그 파일 유실
- summary 파일 유실
- 로그 삭제 성공
- 로그 삭제 중 DB 실패

즉 이전처럼 단순히 "500이 났다"에서 끝나는 것이 아니라,
서버 로그에서

- 어느 동작이었는지
- 어떤 log_id / filename / storage / site였는지

를 더 빨리 좁힐 수 있게 됐다.

### 6. HTTP 요청 자체도 운영 로그로 남는다

애플리케이션 전역 HTTP 로깅을 넣어서

- method
- path
- status code
- client
- duration_ms

를 기록한다.

즉 장애가 "기능 내부 문제"인지,
"특정 API가 계속 실패하는 문제"인지,
"느린 요청이 누적되는 문제"인지
기본 단서가 더 잘 남게 됐다.

### 7. 테스트로 운영성 기능을 검증할 수 있게 됐다

이번 P6에서 운영성 기능도 테스트에 포함했다.

- `/health` 응답 구조 검증
- 관리자 integrity 보고서 검증
- summary 유실 / orphan 파일 탐지 검증

즉 운영용 기능도 이제 수동 확인만 아니라 자동 검증 가능한 범위에 들어왔다.

### 8. 결과적으로 어디를 보면 되는지가 명확해졌다

이번 P6 이후 운영자가 상태를 확인하는 기본 순서는 다음처럼 정리된다.

1. `/health` 로 앱/DB/업로드 경로/스키마 상태 확인
2. 서버 로그로 어느 요청/어느 기능에서 실패했는지 확인
3. `/admin/operations/integrity` 또는 `scripts/check_integrity.py` 로 파일-DB 정합성 점검

즉 "브라우저에서 이것저것 눌러보며 추측"하던 상태에서
"정해진 확인 지점 3곳으로 빠르게 좁히는 상태"로 한 단계 올라왔다.

## 8. 현재 코드 기준 P6 필요성을 가장 잘 보여주는 파일

### [app/api/web_routes.py](/root/2026_project/app/api/web_routes.py:1)

health나 기본 상태 확인 엔드포인트는 운영자가 가장 먼저 보는 진입점이 될 수 있다.

즉 여기는 단순 응답이 아니라,
운영 판단에 도움이 되는 정보를 줄 수 있는 후보 지점이다.

### [app/services/log_service.py](/root/2026_project/app/services/log_service.py:1)

업로드/summary/삭제 흐름의 핵심이 이 파일에 있기 때문에,
운영성 강화에서도 여기가 중요하다.

- 실패 로그
- cleanup 로그
- summary 누락 감지

같은 운영 정보는 결국 이 계층과 연결된다.

### [deploy/run.sh](/root/2026_project/deploy/run.sh:1)

실행 경로는 운영 기준의 진입점이기 때문에,

- migration 적용
- 환경 검증
- 실행 로그

등을 더 분명하게 남길 수 있는 지점이다.

## 9. 최종 판단

P6를 지금 해야 하는 이유는 명확하다.

- 구조는 많이 정리됐고
- 이제는 "문제가 생겼을 때 얼마나 빨리 볼 수 있느냐"가 다음 병목이다.

이 상태에서 P6를 미루면:

- 운영 중 이상을 늦게 발견하고
- 장애 대응 시간이 길어지고
- 사람 기억과 수동 점검에 계속 의존하게 된다.

따라서 P6는 단순 보조 작업이 아니라,

- 현재 시스템을 실제 운영 가능한 도구로 더 끌어올리는 단계
