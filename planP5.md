이 문서는 "P5. 기능별 UX 개선"을 왜 지금 해야 하는지에 집중해서 정리한 문서다.  
핵심은 예쁜 화면을 만드는 것이 아니라, 이미 구현된 기능을 운영자가 더 빠르고 덜 피곤하게 사용할 수 있게 만들어 실제 사용성을 끌어올리는 데 있다.

---

## 1. 결론

P5를 지금 해야 하는 이유는, 현재 시스템이

- 보안 경계(P0)
- 스키마 관리(P1)
- 저장 정합성(P2)
- 테스트 체계(P3)
- 프론트 구조 정리(P4)

까지 어느 정도 기반을 갖춘 상태라서,
이제부터는 "기능이 있느냐"보다 "기능을 얼마나 빠르게 쓸 수 있느냐"가 다음 병목이 되기 때문이다.

현재 기능 자체는 꽤 많다.

- 로그 업로드
- 사이트별 로그 조회
- 원본/summary 보기
- 삭제 요청 승인
- 수정 요청 게시판
- 버그 게시판
- 회원/관리자 화면

하지만 실제 운영 사용성 관점에서는 여전히 아쉬운 부분이 있다.

- 로그 목록 검색 부재
- 정렬/필터 부재
- 페이지네이션 부재
- summary 섹션 전환 가독성 한계
- 게시판 관리 흐름 단순함

즉 지금 문제는 "없는 기능"보다
"이미 있는 기능을 빠르게 찾고 처리하기 어려운 점"에 가깝다.

그래서 P5의 목적은 다음 한 줄로 요약할 수 있다.

- 현재 기능을 운영자 중심 사용 흐름에 맞게 다듬어, 탐색 속도와 처리 속도를 실제로 올리는 것

즉 P5는 cosmetic 작업이 아니라,

- 탐색 비용 절감
- 반복 작업 피로 감소
- 운영 처리 속도 향상

을 위한 단계다.

---

## 2. 현재 UX가 왜 병목이 되는가

### 2.1 로그 목록이 커질수록 탐색 비용이 커진다

[research.md](/root/2026_project/research.md:707) 와 [research2.md](/root/2026_project/research2.md:356) 에서도 드러나듯,
현재 로그 목록은 검색/정렬/페이지네이션이 없다.

이 구조의 문제:

- 로그가 조금만 많아져도 원하는 항목을 찾기 어려움
- 사이트별로 쌓이는 로그를 빠르게 비교하기 어려움
- 최근 로그/오래된 로그/특정 이름 패턴 로그를 찾는 데 시간이 걸림

즉 지금은 데이터가 늘어날수록 기능은 그대로여도 UX 품질은 떨어지는 구조다.

---

### 2.2 summary는 유용하지만, 빠르게 훑기엔 아직 거칠다

현재 summary는 기능적으로 잘 동작한다.

- overview
- 섹션별 상세
- 특이사항
- Event log

까지 볼 수 있다.

하지만 운영 관점에서는 아직 다음이 아쉽다.

- overview와 상세 섹션 간 시선 이동이 잦음
- 큰 텍스트 영역을 훑는 비용이 큼
- Event log의 가독성과 탐색성이 더 개선될 여지가 있음

즉 summary는 "데이터는 있다" 수준은 넘었지만,
"빠르게 판단하기 좋은 화면"으로는 더 다듬을 수 있다.

---

### 2.3 게시판도 기능은 있으나 운영 흐름은 아직 단순하다

수정 요청 게시판과 버그 게시판은 이미 동작한다.

하지만 운영상 자주 부딪히는 문제는 다음일 수 있다.

- 원하는 글을 빨리 찾기 어려움
- 상태/우선순위/담당 흐름이 약함
- 목록이 쌓일수록 관리 비용이 증가

즉 게시판도 지금은 "등록/수정/삭제가 된다" 수준에 가깝고,
"운영용 작업 큐"로는 아직 덜 다듬어진 상태다.

---

## 3. 왜 지금 P5인가

### 3.1 P1~P4가 끝나서 이제 UX를 손댈 기반이 생겼다

지금까지는 구조 안정화가 우선이었다.

- P1 이전에는 스키마 관리가 흔들렸고
- P2 이전에는 로그/summary 저장 구조가 약했고
- P3 이전에는 회귀 검증이 약했고
- P4 이전에는 프론트 구조 자체가 너무 비대했다

이 상태에서 UX 개선을 먼저 하면,
화면은 좋아져도 기반 구조가 흔들려서 유지보수 비용이 커질 수 있었다.

지금은 그 반대다.

- API 계약이 더 안정적이고
- 테스트도 있고
- 프론트 구조도 어느 정도 나뉘었다

즉 이제 UX를 다듬어도 비교적 안전하다.

---

### 3.2 다음 단계의 가치가 사용자에게 가장 바로 보인다

P1~P4는 주로 기반 정리였다.

운영에는 중요하지만, 사용자 눈에는 변화가 직접 크게 보이지 않을 수도 있다.

반면 P5는 바로 체감된다.

- 더 빨리 찾고
- 더 적게 클릭하고
- 더 쉽게 상태를 파악하고
- 더 짧은 시간에 일을 처리할 수 있게 된다

즉 지금은 기반 정리 이후,
그 효과를 실제 사용자 경험으로 전환하는 단계라고 볼 수 있다.

---

## 4. 지금 미루면 생기는 문제

### 4.1 기능은 많아도 "불편한 도구"로 남을 수 있다

지금 구조로도 기능은 돌아간다.

하지만 사용자가 느끼는 평가는 종종
"무엇을 할 수 있느냐"보다
"얼마나 쉽게 할 수 있느냐"에서 갈린다.

UX 개선을 미루면:

- 로그가 쌓일수록 찾기 힘들고
- 게시판이 쌓일수록 관리가 번거롭고
- summary는 계속 읽기 피곤한 화면으로 남는다

즉 기능 밀도 대비 사용 만족도가 낮아질 수 있다.

---

### 4.2 데이터가 늘수록 불편함이 더 커진다

지금의 UX 문제는 데이터가 적을 땐 덜 아프지만,
데이터가 늘수록 더 커지는 유형이 많다.

대표적으로:

- 목록 검색 부재
- 페이지네이션 부재
- 상태 필터 약함

즉 지금 손보지 않으면 나중에는 단순 개선이 아니라,
"운영 규모 증가로 인한 필수 개편"이 될 수 있다.

---

## 5. P5가 해결하려는 핵심 문제

P5가 해결하려는 핵심은 아래 다섯 가지다.

1. 로그 목록 탐색 속도 저하
2. summary 읽기/비교 비용
3. 게시판 관리 흐름의 단순함
4. 데이터 증가에 비례해 커지는 클릭 수와 시선 이동
5. 기능은 있지만 처리 시간이 긴 운영 UX

즉 P5의 본질은:

- "현재 기능을 운영 효율 관점에서 다시 다듬는 것"

이다.

---

## 6. P5가 끝나면 얻는 것

### 운영 측면

- 원하는 로그를 더 빨리 찾을 수 있음
- 요약/상세 판단 속도가 빨라짐
- 게시판성 작업의 처리 속도가 올라감

### 개발 측면

- 프론트 구조를 이미 나눈 상태라 기능별 UX 개선도 더 국소적으로 가능
- API 확장 포인트가 더 명확해짐

### 사용자 경험 측면

- "기능은 있는데 불편한 도구"에서 벗어남
- 반복 사용 피로가 줄어듦
- 운영자 만족도가 직접 올라갈 가능성이 큼

---

## 7. P5는 무엇을 하지 않는가

P5 범위를 분명히 할 필요가 있다.

P5는 아래를 주로 다룬다.

- 로그 목록 검색/정렬/필터/페이지네이션
- summary 보기 개선
- 게시판 UX 개선

반면 아래는 P5의 핵심 목표가 아니다.

- 백엔드 저장 구조 재설계
- 인증 정책 변경
- 프론트 구조 대규모 재리팩토링
- 운영 스크립트/헬스체크 구축

즉 P5는 "구조 안정화 이후의 사용성 개선 단계"다.

## 8. 현재 코드 기준 P5 필요성을 가장 잘 보여주는 파일

### [app/static/app.js](/root/2026_project/app/static/app.js:1)

현재 목록 렌더링, summary 렌더링, 게시판 렌더링이 모두 이 파일에 들어 있고,
검색/정렬/필터/페이지네이션 요구도 결국 여기와 직접 연결된다.

즉 P5는 프론트 UX를 다루지만,
실제로는 API와 상태 구조를 함께 건드리는 작업이 될 가능성이 높다.

### [app/services/log_service.py](/root/2026_project/app/services/log_service.py:1)

로그 목록 검색/정렬/페이지네이션을 제대로 하려면
프론트만이 아니라 서버 목록 API도 같이 바뀔 수 있다.

즉 P5는 화면 개선이지만,
로그 목록 API 확장과 연결될 가능성이 크다.

### [app/services/board_service.py](/root/2026_project/app/services/board_service.py:1)

게시판 UX를 더 운영 친화적으로 바꾸려면,
필터/상태/우선순위/정렬 규칙이 서비스 로직까지 이어질 수 있다.

## 9. 최종 판단

P5를 지금 해야 하는 이유는 명확하다.

- 구조 안정화 작업은 P1~P4에서 상당 부분 끝났고
- 이제부터는 "사용 속도"가 다음 병목이 된다.

이 상태에서 P5를 미루면:

- 기능은 많지만 계속 불편한 운영 도구로 남고
- 데이터가 쌓일수록 불편함이 커지고
- 사용성 문제를 나중에 더 큰 비용으로 해결하게 될 가능성이 있다.

따라서 P5는 단순 화면 다듬기가 아니라,

- 기반 정리 이후 실제 체감 품질을 올리는 단계

라고 보는 것이 맞다.

한 줄로 요약하면:

P5는 지금 프로젝트에서 "기능은 되지만 처리 속도가 느린 운영 도구"를 끝내고,  
"운영자가 빠르게 찾고 빠르게 처리할 수 있는 도구"로 넘어가기 위해 필요한 단계다.
